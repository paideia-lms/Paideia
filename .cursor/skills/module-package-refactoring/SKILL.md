# Module-to-Package Refactoring

Guidance for extracting a Paideia backend module into a standalone workspace package that can run independently (own Payload config, migrations, tests).

## When to Use

- Creating `packages/module-user`, `packages/module-infrastructure`, or similar self-contained packages
- Migrating code from `packages/paideia-backend/src/modules/{name}/` to `packages/module-{name}/`
- Making a module runnable in isolation (own `payload.config.ts`, `migrate:fresh`, tests)

## Package Structure

```
packages/module-{name}/
├── src/
│   ├── index.ts              # Single point of export; exposes only class + namespace
│   ├── payload.config.ts     # Mock/minimal Payload config for isolated schema generation
│   ├── payload-types.ts      # Generated: bun run generate:payload
│   ├── payload-generated-schema.ts  # Generated, add // @ts-nocheck
│   ├── collections/          # Payload collection configs
│   ├── services/             # try* functions (Result-based)
│   ├── api/                  # oRPC endpoints
│   ├── orpc/
│   │   └── context.ts        # OrpcContext (payload, s3Client, user, req)
│   ├── errors.ts             # Module-specific errors
│   ├── utils/
│   │   └── constants.ts      # devConstants, MOCK_INFINITY, etc.
│   ├── seeding/              # SeedBuilder, predefined data
│   ├── migrations/           # Payload migrations
│   └── tests/
├── package.json
├── tsconfig.json
└── README.md
```

### Export rules

- **index.ts** is always the single point of export for the package.
- **Only a class and a namespace** can be exposed. Nothing else (no bare `export { x }`, no exported functions/constants at top level).
- The class implements the module facade; the namespace holds shared types.

### moduleName and dependencies

Use `packageJson` so values stay in sync with `package.json` and avoid mistakes:

```typescript
import packageJson from "../package.json";

export class NoteModule {
    public static readonly moduleName = packageJson.name;
    public static readonly dependencies = Object.keys(packageJson.dependencies);
    // ...
}
```

### Mock payload.config.ts

Each module has a **mock** `payload.config.ts` so that a minimal db schema can be generated for the module. This lets tests run in isolation without the full app.

## Path Import Fixes

When moving from paideia-backend, replace internal paths with local package paths:

| Old (paideia-backend) | New (module package) |
|----------------------|----------------------|
| `server/collections` | `./collections/{name}` |
| `server/errors` | `../errors` |
| `payload.config` | `./payload.config` (tsconfig path) |
| `payload-types` | `./payload-types` (tsconfig path) |
| `payload-generated-schema` | `../payload-generated-schema` |
| `../utils/constants` | `../utils/constants` (relative from service) |

**tsconfig.json** paths:

```json
{
  "paths": {
    "payload.config": ["./src/payload.config.ts"],
    "payload-types": ["./src/payload-types.ts"],
    "payload-generated-schema": ["./src/payload-generated-schema.ts"]
  },
  "exclude": ["node_modules", "src/payload-generated-schema.ts"]
}
```

## Type Fixes for Standalone Packages

### 1. payload-generated-schema.ts

Add at top (first line):

```typescript
// @ts-nocheck - Generated by Payload, has circular refs and schema conflicts
```

Generated schema has circular refs (users ↔ media) and `DatabaseSchema` conflicts. Excluding from typecheck still pulls it in via imports; `@ts-nocheck` suppresses errors.

### 2. envVars (string | undefined)

When using `envVars.S3_BUCKET.value` where `string` is required:

```typescript
Bucket: envVars.S3_BUCKET.value ?? envVars.S3_BUCKET.default ?? "",
```

### 3. Drizzle dynamic schema columns (tryFindMediaUsages)

For `eq()` with dynamic join table columns, use `Parameters<typeof eq>`:

```typescript
eq(parentTable.id, joinTable._parentID as Parameters<typeof eq>[1])
eq(joinTable.mediaID as Parameters<typeof eq>[0], mediaId)
```

### 4. Payload TaskConfig (module-infrastructure)

TaskConfig has known type incompatibility with outputSchema (Payload #13681). Use:

```typescript
handler: async ({ req }: { req: PayloadRequest }) => { ... },
} as unknown as TaskConfig;
```

## Dependencies

Add to `package.json`:

- `@paideia/module-infrastructure` (workspace:*)
- `@paideia/shared` (workspace:*)
- `payload`, `@payloadcms/db-postgres`, `@payloadcms/storage-s3`
- `typescript-result`, `zod`, `es-toolkit`
- Any module-specific deps (e.g. `search-query-parser` for user search)

## Test Data Isolation

**Critical**: Test search/filter data must not collide with seed data.

Example: Searching for `"Manager"` matches both a test user (Bob Manager) and a seed user (Content Manager). Use unique values:

```typescript
// ❌ Collides with seed (Content Manager)
{ lastName: "Manager" }

// ✅ Unique - not in predefined-user-seed-data
{ lastName: "Coordinator" }
```

Always check `predefined-*-seed-data.ts` before writing search/filter assertions.

## Collection Hooks and Transaction Visibility

When migrating collections with hooks (`beforeValidate`, `beforeChange`, etc.), ensure ALL Payload operations inside hooks pass `req` to maintain transaction context. This is critical for SeedBuilder-based tests.

```typescript
// ❌ Breaks inside SeedBuilder transactions
const parent = await req.payload.findByID({ collection: "groups", id: data.parent });

// ✅ Maintains transaction context
const parent = await req.payload.findByID({ collection: "groups", id: data.parent, req });
```

See incident: `release-notes/incidents/2026-03-09-groups-hook-transaction-visibility.md`
